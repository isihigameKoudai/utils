# Idea Refine

## ゴール

- 1 案目への飛びつきを防ぎ、前提の異なる複数案を比較した上で 1 案に収束する
- 「採用した案の根拠」と「捨てた案の理由」が両方残った状態で終了する
- 比較は感想ではなく、先に固定した評価軸に対して行う

## 手順

### 1. 前提を確認する

問題文を 1 文（How Might We 形式）で書き直し、「誰のため・成功条件・制約」が不明なら 1–3 問で確認する。要求自体が固まっていない場合は `.agents/skills/interview-me/SKILL.md` に戻る。

### 2. 発散 — 前提を変えて 3–5 案つくる

- 各案は **異なる前提を 1 つ変える**こと。同じ案の表記ゆれを並べない
  - 例: 置き場所を変える（`utils/` の framework 非依存ユーティリティ vs `src/features/` の機能実装）
  - 例: 依存を変える（新規ライブラリ導入 vs 自前実装 vs 既存 utils の拡張）
  - 例: スコープを変える（10x シンプルな最小版 vs フル機能版）
- このリポジトリで動く案にする: 既存の `utils/` `src/` を Grep / Read で確認し、各案に既存ファイル・パターンへの参照を添える。アーキテクチャ制約（routes → features → components → utils の単方向依存、React 19 + TypeScript strict + Vite 8、pnpm のみ）に反する案は最初から出さない
- 3–5 案で止める。20 案の浅いリストより、理由のある 3 案

### 3. 収束(1) — 評価軸を先に固定する

**案を比較・削除する前に**、評価軸を 3–5 個決めてユーザーに提示する。例:

- 実装コスト（変更ファイル数・新規依存の有無）
- 既存アーキテクチャとの整合（依存方向・`utils/` の再利用性を壊さないか）
- 検証可能性（`pnpm vitest run` / `pnpm run build` で確認できるか、ブラウザ確認が要るか）
- 可逆性（やめたくなったとき戻せるか）

軸はアイデアに合わせて選ぶが、決定後に都合よく追加・削除しない。

### 4. 収束(2) — 比較して 1 案に絞る

- 全案を全軸で評価し、比較表にする
- 各案の **隠れた仮定**（検証していないが真だと賭けていること）を明示する
- 弱い案には率直に弱いと言う。yes マシンにならない
- 推奨案 1 つと、捨てた案それぞれの理由を書く

### 5. 確定と引き継ぎ

- ユーザーが推奨案を確認したら、出力フォーマットの成果物を提示する
- 永続化したい場合のみ `doc/plans/<topic>.md` に保存する（保存はユーザー確認後のみ）
- 後続: 採用案を仕様に落とす → `.agents/skills/spec-driven-development/SKILL.md`

## Hard rules

- **評価軸を決める前に案を絞らない**。直感での足切りは発散フェーズを無意味にする
- **1 案目に飛びつかない**。最低 3 案、各案は前提が異なること
- 比較表なしで「これがおすすめです」と言わない。捨てた案の理由を必ず残す
- 既存コードを無視した案を出さない（各案に関連する既存ファイル・パターンを添える）
- 隠れた仮定をリストせずに収束しない
- 出力本文は日本語。識別子・コマンド・パスは原文のまま

## 出力フォーマット

````
IDEA REFINE
===========

問題文: <How Might We 形式 1 文>

## 比較表

| 評価軸 | 案A: <名前> | 案B: <名前> | 案C: <名前> |
|---|---|---|---|
| 実装コスト | ... | ... | ... |
| アーキテクチャ整合 | ... | ... | ... |
| 検証可能性 | ... | ... | ... |
| 可逆性 | ... | ... | ... |

## 推奨案: <案X>

- 根拠: <評価軸に基づく理由 2–3 行>
- 隠れた仮定: <真だと賭けていること + 検証方法>
- 関連する既存コード: <path>

## 捨てた案と理由

- 案Y: <どの軸で劣ったか 1 行>
- 案Z: <どの軸で劣ったか 1 行>

## 未解決の質問

- <仕様化の前に決めるべきこと。なければ「なし」>
````

## 完了チェックリスト

- [ ] 前提の異なる案を 3–5 個出した（表記ゆれの水増しがない）
- [ ] 評価軸を **比較の前に** 固定し、ユーザーに提示した
- [ ] 全案 × 全軸の比較表がある
- [ ] 推奨案に根拠と隠れた仮定、捨てた案に理由が書かれている
- [ ] 各案が既存の `utils/` / `src/` の実態に接地している
- [ ] ユーザーが推奨案を確認した（保存する場合は確認後に `doc/plans/<topic>.md`）

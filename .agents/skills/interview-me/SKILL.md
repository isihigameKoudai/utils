# Interview Me

## ゴール

- 計画・仕様・コードを書く前に、ユーザーが「口にした要求」ではなく「実際に欲しいもの」を言語化する
- 目的 / スコープ / 非スコープ / 制約 / 検証方法が明文化され、ユーザーの明示的な「はい」を得た状態で終了する
- AGENTS.md の方針（確信度 80% 未満なら質問する・曖昧/セキュリティセンシティブな要件はエスカレーション）をこのスキルで実行する

## 手順

### 1. 仮説と確信度を先に出す

質問を始める前に、現時点の読みを 1 文 + 確信度で提示する。

```
仮説: <ユーザーが本当に欲しいものの 1 文要約>
確信度: ~40% — 不足: 誰のためか / 「速く」の具体値 / 完了の確認方法
```

確信度が 80% 未満なら、何が欠けているかを同じ行に必ず書く（ユーザーが埋めるべき穴を明示するため）。

### 2. 一度に 1–3 問、選択肢つきで聞く

- 1 ターンの質問は **1–3 問まで**。4 問以上のリスト爆撃をしない
- 各質問には **選択肢（A/B/C）または自分の推測（GUESS）** を添えて認知負荷を下げる。ゼロから答えさせるより、間違った推測に反応させる方が速い
- 次の質問は前の答えから導出する。最初に決めた質問リストを機械的に消化しない

```
Q: この hook は src/ の feature 専用ですか、それとも utils/ に置く再利用前提ですか？
   A) feature 専用（src/features/ 配下） B) 再利用前提（utils/ 配下、React 非依存も検討）
GUESS: B。過去の追加パターンは utils/ 起点が多いため
```

### 3. 「建前回答」を検出したら掘る

「スケーラブルに」「モダンに」「ちゃんとした設計で」のような、具体的な結果を伴わない回答が出たら、次の 1 問で掘る。

> 「誰にも正当化しなくていいとしたら、本当はどうしたいですか？」

### 4. 終了条件を判定する

次の 4 点が言語化できたら質問をやめる。できていないのに質問を続けても確信度が上がらない場合は、その旨を伝えて一段抽象的な論点に戻る。

1. **目的**（何のために）
2. **スコープと非スコープ**（やること・やらないこと）
3. **制約**（技術・互換性・時間）
4. **検証方法**（`pnpm vitest run` / `pnpm run build` / ブラウザでの操作手順など、実行可能な確認手段）

### 5. 要求サマリを書き戻し、明示的な「はい」を得る

下記フォーマットでユーザーの言葉を使って書き戻す。「お任せします」「いい感じで」は yes ではない — 2 択に変換して再確認する。修正されたら反映して再提示し、明示的な「はい」までループする。

### 6. 後続スキルへ引き継ぐ

- スコープの取り方自体に複数案がある → `.agents/skills/idea-refine/SKILL.md`
- 要求が具体的に固まった → `.agents/skills/spec-driven-development/SKILL.md`

## Hard rules

- 1 ターンの質問は 1–3 問まで。推測か選択肢を添えない質問をしない
- 「お任せ」「sounds good」「沈黙」を確認とみなさない。明示的な「はい」のみがゲート
- 確認前に仕様・計画・コードを書き始めない
- 要求サマリから「非スコープ」を省略しない（認識ズレの半分は「やらないこと」の暗黙の不一致）
- 非対話コンテキスト（CI / 定期実行 / 自律ループ）では起動しない。曖昧さはブロッカーとして報告する
- 対話・出力は日本語。識別子・コマンド・パスは原文のまま

## 出力フォーマット

```
要求サマリ
==========
目的:         <1 行 — 何のために>
スコープ:     <やること>
非スコープ:   <やらないこと>
制約:         <例: utils/ は再利用前提で React 非依存に保つ / pnpm のみ使用 / TypeScript strict>
検証方法:     <例: pnpm vitest run が green / pnpm run build が通る / /xxx ルートで操作 Y ができる>
未解決の質問: <残った論点。なければ「なし」>
```

## 完了チェックリスト

- [ ] 初手で仮説 + 確信度（80% 未満なら不足理由つき）を提示した
- [ ] 各ターン 1–3 問、選択肢または推測つきで質問した
- [ ] 建前回答（バズワード回答）には「本当はどうしたいか」で掘り直した
- [ ] 目的・スコープ・非スコープ・制約・検証方法がすべて言語化された
- [ ] 要求サマリに対してユーザーの明示的な「はい」を得た
- [ ] 後続スキルへの引き継ぎは、元の曖昧な依頼ではなく確認済みサマリに基づいている

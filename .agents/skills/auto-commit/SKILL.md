# Auto Commit

## ゴール

- 現在の作業ツリーやステージの差分を分析し、リポジトリのルールに沿った適切なコミットメッセージを生成する
- 安全にファイルをステージし、コミットを完了させる

## 手順

### 1. 差分とステータスの確認

まずは現在の状態を確認する。

```bash
git status
git diff --cached
git diff
```

### 2. 静的解析・テストの実行

コミット前に、コードに問題がないか静的解析（lint）等を行う。

```bash
pnpm run lint
pnpm run format
```

### 3. ステージングの実施

未ステージの変更があり、それをコミットに含めるべき場合は明示的にステージする。
※ `git add .` や `git add -A` は使わず、対象ファイルを明示すること。

```bash
git add <対象ファイルパス>
```

### 4. コミットメッセージの生成と提示

差分内容から「何を変更したか」を抽出し、コミットメッセージを作成する。
**作成した最終的なコミットメッセージは、コミットを実行する前に必ずユーザーに提示する（出力する）こと。**

このリポジトリのルール(`git-workflow-and-versioning`)に従い、以下のフォーマットとする。
- **短い英語命令形** (例: `add xxx`, `fix yyy`, `refactor zzz`)
- 長文の body や `feat:` などのプレフィックスは使用しない

### 5. コミットの実行と確認

生成したメッセージでコミットを実行する。

```bash
git commit -m "<生成したメッセージ>"
```

コミット後、husky などの pre-commit hook により内容が変更されていないか確認する。

```bash
git status
git show --stat HEAD
```

## Hard rules

- 変更内容（diff）を必ず読み、変更の意図を正確に捉えたメッセージにすること
- 秘密情報（APIキーやパスワードなど）が diff に含まれていないか必ず確認すること
- リポジトリのルールに則り、コミットメッセージは必ず**英語の命令形**とすること
- `git add .` を使わず、意図したファイルのみをステージすること
- **コミットを実行する前に、静的解析（lint）を通し、最終的なコミットメッセージを提示すること**

## 完了チェックリスト

- [ ] `git diff` および `git status` で差分を正確に把握した
- [ ] 静的解析（`pnpm run lint`等）を実行し、エラーがないことを確認した
- [ ] 意図したファイルだけがステージされていることを確認した
- [ ] 英語の短い命令形でコミットメッセージを生成し、ユーザーに提示した
- [ ] `git commit` が成功し、フックによる意図しない残差分がないことを確認した

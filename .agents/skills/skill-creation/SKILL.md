# Skill Creation (カスタムスキルの作成・評価・最適化)

## ゴール

- プロジェクト固有の知識、手順、およびGotchasを正確に捉えた、500行・5000トークン未満の高品質な `SKILL.md` を作成する
- エージェントが実行環境で正しく動く、非インタラクティブで安全な再利用スクリプトを `scripts/` に適切にバンドルする
- 作成したスキルについて、テストケース、アサーション、段階的な自動評価ループを定義し、トリガー精度と出力品質の双方を定量的に検証・改善する

## 手順

### 1. 本物の専門知識（Expertise）の注入

汎用的なLLMの知識のみに頼らず、プロジェクト固有の事実をスキルに落とし込みます。

- **作業実績からの抽出**: エージェントとの対話で成功した一連のコマンド、直面したエラー、修正手順、およびデータ構造を再利用可能なパターンとして抽出する。
- **チーム資産の活用**: 既存のスタイルガイド、API仕様（OpenAPI/JSON Schema）、インシデントランブック、テスト結果、および過去のPR差分を統合する。

### 2. 文脈予算（Context Budget）の管理

スキルがロードされた際のコンテキスト肥大化を防ぎ、エージェントの注意を惹きつける設計にします。

- **エージェントが知っていることは書かない**: PDFやHTTP、一般的なデータベースマイグレーションの仕組みなど、基礎知識の説明は徹底的に排除する。スキル固有・プロジェクト固有の制約に焦点を絞る。
- **段階的開示（Progressive Disclosure）の活用**: リファレンスや長大なスキーマ、巨大なテンプレートは `SKILL.md` に直接書かず、`references/` などの別ディレクトリに配置する。`SKILL.md` からは「非200エラーが発生した場合は `references/api-errors.md` を読め」のように動線（トリガー条件）のみを指示する。

### 3. 制御レベル（Calibrating Control）の調整とデフォルトの提示

タスクの脆弱性（壊れやすさ）に応じて、指示の厳密さを調整します。

- **自由度を持たせる記述**: 結果のバリエーションが許容される領域（コードレビュー、リファクタリングの視点など）では、手順を固定せず「目的」や「観点」を解説する。
- **厳密に強制する記述**: マイグレーションの順序やビルド/リリースフローなど、順序が命取りになる手順は、代替手段のない厳格な手順（チェックリストや特定コマンド）として書く。
- **選択肢ではなく「デフォルト」を1つ示す**: 複数のアプローチがある場合は「X, Y, Zが使える」と並べるのではなく、最良の「デフォルト」を推奨し、例外時のみ代替策を提示する。

### 4. 効果的な指示（Instruction）パターンの実装

以下の5つの指示設計パターンを適宜スキルに組み込みます。

- **Gotchas（落とし穴）セクション**:
  エージェントが暗黙の仮定で失敗しやすい「仕様の罠」や「非直感的な事実」をリスト化して、実行前に必ず読ませる。
  _例: 「テーブル A は論理削除を使用しているため、常に `deleted_at IS NULL` を付与すること。」_
- **出力用テンプレート**:
  特定のフォーマットを期待する場合は、散文で説明するのではなく、MarkdownやJSONのプレースホルダー付きテンプレートコードブロックを直接提示する。
- **マルチステップ・チェックリスト**:
  依存関係や検証ゲートを伴う作業では、進行状況を管理するための `- [ ]` チェックリストを明示する。
- **検証（Validation）ループ**:
  「変更を実行」→「検証スクリプトやチェックリストによる確認」→「エラーがあれば修正」→「パスするまで繰り返す」という検証ループを手順化する。
- **計画・検証・実行（Plan-Validate-Execute）**:
  破壊的またはバッチ処理を行うスキルでは、実行前に中間計画ファイル（例: `plan.json`）を書き出させ、検証スクリプトでチェックした後にのみ最終実行コマンドを呼ぶように手順を分割する。

### 5. 自律スクリプト（Scripts）のバンドル

エージェントが同一手順を毎回「自力で発明」するのを防ぐため、再利用可能な検証・実行スクリプトを `scripts/` にバンドルします。

- **自己完結型（Self-contained）の依存宣言**:
  スクリプトを単一ファイルで動かせるよう、インライン依存宣言（Python PEP 723、Denoの `npm:`/`jsr:` インポート、Ruby `bundler/inline` 等）を使用する。
- **絶対厳守の制約**:
  - **対話型プロンプトの完全排除**: TTY入力、パスワードダイアログ、決定確認プロンプト（`Y/N`）を一切含めない。すべての入力はコマンドライン引数、環境変数、または stdin から受け取り、不足時は `--help` と共に非ゼロコードで即終了する。
  - **`--help` オプションの完備**: エージェントが実行前に引数や使用例を学べるよう、簡潔な説明・オプション一覧・具体的な使用例を含むヘルプを実装する。
  - **有益なエラーログ**: エージェントがエラー発生時に自律修正できるよう、何が間違っており（渡された不正な値）、何を期待していたか（有効な選択肢）を具体的に出力する。
  - **構造化された出力**: stderrに診断情報を、stdoutにJSON/CSVなどの構造化データを出力し、エージェントや他のツール（`jq`等）がプログラム的に処理しやすくする。
  - **冪等性（Idempotency）と安全なデフォルト**: 再試行に耐えうる「存在しなければ作成」といった挙動にし、破壊的操作には `--confirm` などのオプションを要求する。

### 6. 評価（Evaluation）とトリガー最適化

スキル作成後、それが確実に意図したタイミングでトリガーされ、高品質な出力を生むか評価します。

- **トリガー定義（description）の最適化**:
  - `SKILL.md` のフロントマター `description` は1024文字以内で、命令形（"Use this skill when..." / "〜するときに使用する。"）で書く。
  - **評価用クエリ（eval_queries.json）の作成**:
    - スキルを呼ぶべきクエリ（should_trigger: true）を8-10個（カジュアルな表現、詳細なもの、タイポありなど）
    - スキルを呼ぶべきではない「ニアミス」クエリ（should_not_trigger: false）を8-10個（キーワードが重複しているが無関係な作業）
  - **評価と分割（Train/Val split）**:
    - クエリを Train 60% / Validation 40% に分割し、Trainの失敗を元に `description` を修正し、 Validation で汎化性能を確認する。
    - 各クエリについて複数回（3回以上）実行し、トリガー率（Trigger Rate）がしきい値（例: 0.5）を超えているか検証する。
- **出力品質（Output Quality）の自動アサーション**:
  - **テストケース設計（evals/evals.json）**: 現実的なプロンプト、インプットファイル、および期待される結果を記述する。
  - **アサーションの定義**: 機械的に検証可能なアサーション（"出力ファイルが有効なJSONであること"）や視覚的/客観的なチェック項目（"グラフに軸ラベルがあること"）を定義する。
  - **スコアリングと改善**: スキルなし（または過去のバージョン）をベンチマーク対照として実行し、トークン数、処理時間、アサーションのパス率（Mean Pass Rate）の差分（`delta`）を計算して品質を数値化する。

## Hard rules

- **対話型（Interactive）スクリプトの作成禁止**: ユーザー入力を待つスクリプトはエージェント環境をハングさせるため絶対に使用しない。
- **説明（description）の文字数制限**: フロントマターの `description` は必ず1024文字以内に収める。
- **具体的ではないGotchasの禁止**: 「エラーに注意すること」のような抽象的な記述は避け、「API Aは非推奨となりBが優先される」のように具体的かつ検証可能な事実を書く。
- **依存ライブラリのバージョン固定**: スクリプト内のインライン依存や、コマンドで使うパッケージ（例: `npx eslint@9`）には必ずバージョン指定を含める。

## 完了チェックリスト

- [ ] プロジェクト固有のGotchasや手順が、具体的に記述されている（エージェントが暗黙で知っている内容は排除されている）
- [ ] バンドルされたスクリプト（`scripts/`）は、対話プロンプトなしで動き、`--help` を完備している
- [ ] スキーマ、大容量のテンプレートなどは `references/` や `assets/` に外部化され、 progressive disclosure 設計になっている
- [ ] フロントマターの `description` は1024文字以内の命令形で、トリガーすべき状況が網羅されている
- [ ] スキルのトリガー精度と出力品質を検証するための評価用クエリ、テストケース、およびアサーションが `evals/` 以下に準備されている

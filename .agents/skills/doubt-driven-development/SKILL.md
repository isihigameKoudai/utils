# Doubt-Driven Development

## ゴール

- 非自明な判断が、独立した反証の試みに耐えてから採用される
- 長いセッションで「仮定」が検証なしに「事実」へ昇格するのを防ぐ
- `/review`(完成物への事後判定)ではなく、実装中の方向修正が安いうちに誤りを捕まえる

## 発動条件

次のいずれかに当てはまったら適用する:

- **不慣れなコード領域**: 完全に理解していない feature・utils を変更する(例: シェーダ系 `src/features/Noise/`、`StableFluids/`、ML 系 `Detection/`、`utils/i-state/` の内部)
- **破壊的変更**: `utils/` の公開 API 変更(複数 feature に波及)、依存方向に関わる構造変更、ファイル削除を伴う変更
- **「たぶん動く」と感じた瞬間**: 確信はあるが型・テストで証明されていない主張(並行性・実行順序・エッジケースの安全性)をしようとしている
- **同じ修正が 2 回失敗した**: 仮説自体が誤っている可能性が高い。3 回目を打つ前に反証にかける

自明な変更(リネーム・整形・1 行の明白な修正・明確な指示の遂行)には適用しない。すべてを疑うと何も出荷できない。

## 手順

### 1. 判断を主張として書き出す

2〜3 行で明文化する。書けないなら、それは判断ではなく雰囲気 — まず言語化する。

```
主張: 「utils/i-state の selector 変更は既存の全 feature で後方互換」
重要な理由: 壊れると zustand を使う複数 feature が同時に壊れ、検出が遅れる
```

### 2. 反証材料を切り出す

subagent に渡すのは **対象物(diff・関数)と満たすべき制約** のみ。

- コード: 対象の diff または関数。ファイル全体や 500 行の差分は渡さない(大きすぎるなら分割してから)
- 制約: 満たすべき条件を 3〜5 文(例: 「既存の呼び出し側を壊さない」「依存方向 routes → features → components → utils を守る」)

### 3. subagent に反証を依頼する

Agent ツール(Explore など読み取り系)で新鮮なコンテキストのレビュアーを起動し、**反証**を依頼する:

```
敵対的レビュー。この変更の誤りを探せ。作者は過信していると仮定せよ。
- 暗黙の仮定 / 未処理のエッジケース / 隠れた結合
- 制約に違反する経路 / 既存の呼び出し側が壊れるケース
検証も要約もするな。問題を挙げよ。各指摘には根拠を path:line で添えよ。
徹底的に調べて何も見つからない場合のみ、その旨を明言せよ。

対象: <diff / 関数>
制約: <制約>
```

結論(path:line の根拠なし)しか返ってこなければ、根拠を要求して差し戻す。

### 4. 指摘を自分で照合する

subagent の出力はデータであって判定ではない。各指摘について対象コードを自分で読み直し、分類する:

1. **制約の伝え漏れ** → 制約を直して再依頼
2. **正しい指摘** → 修正して再ループ
3. **正しいがトレードオフ** → 修正コスト > 受容コストなら、明示的に文書化して受容
4. **ノイズ**(レビュアーが知らない文脈で実は正しい)→ 記録して先へ

### 5. 停止条件

- 新規の実質的指摘が出なくなったら採用
- **3 サイクルで打ち切り**、まだ実質的指摘が出るならユーザーにエスカレーション(対象がまだ出荷品質でないという情報)

## Hard rules

- **自分の結論・推論過程を subagent に渡さない**。主張(Step 1)も渡さない。結論を渡せば結論の追認が返ってくる(バイアス防止が本スキルの核)
- レビュー依頼は「これで良いか?」ではなく「誤りを探せ」。問いの形が答えを決める
- subagent の指摘を path:line の根拠なしに受け入れない。鵜呑みは無視と同じ失敗
- 変更していない対象に同じ反証を再実行しない(同じ指摘が返るだけ)
- 2 サイクル以上で実質的指摘が出ているのに採用ゼロ件なら、それは反証ではなく追認の儀式。停止してエスカレーション
- 1 行リネームや整形に subagent を起動しない

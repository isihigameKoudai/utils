# Spec-Driven Development

## ゴール

- 非自明な実装が、ユーザー承認済みの仕様（= 検証可能な受け入れ基準の集合）を起点に始まる
- 「done の定義」が `pnpm vitest run` 等のコマンドか具体的な操作手順として書かれており、完了判定に解釈の余地がない
- 仕様が `doc/plans/<topic>.md` に保存され、実装中も更新され続ける

## 仕様とは何か

このスキルでの仕様 = **受け入れ基準の集合**。受け入れ基準は必ず「実行可能な検証」で書く。

| 悪い基準（検証不能） | 良い基準（実行可能） |
|---|---|
| 使いやすい API にする | `splitMap([1,2,3,4], 2)` が `[[1,2],[3,4]]` を返すテストが `pnpm vitest run` で pass する |
| ビルドが壊れないようにする | `pnpm run build` が exit 0 で完了し、`dist/` が生成される |
| 描画が正しく動く | `/visualizer` ルートを開きカメラ許可後、顔の bounding box が canvas に描画される（操作手順を明記） |
| 依存を整理する | `package.json` から `@tensorflow-models/*` が消え、`pnpm vitest run` が green |

## 手順

### 1. 入力を確認する

要求が曖昧（目的・スコープ・検証方法が不明）なら、仕様を書く前に `.agents/skills/interview-me/SKILL.md` で要求サマリを作る。実現方法に複数案があるなら `.agents/skills/idea-refine/SKILL.md` で 1 案に収束させてから来る。

### 2. 仮定を先に晒す

仕様本文を書く前に、自分が埋めようとしている仮定をリストしてユーザーに提示する。

```
仮定:
1. 対象は utils/ に置く再利用ユーティリティ（src/features/ の機能実装ではない）
2. 新規依存パッケージは追加しない
3. 既存の呼び出し元のインターフェースは維持する
→ 違っていれば今指摘してください。確信度 80% 未満の項目は質問に切り替えます（AGENTS.md 方針）
```

セキュリティセンシティブ・UX/API 契約が変わる要件は、仮定で進めずエスカレーションする。

### 3. 仕様を書く

出力フォーマット（下記）に沿って書く。このリポジトリの事実に接地させること:

- 検証コマンドは `pnpm vitest run` / `pnpm run lint` / `pnpm run build` のいずれか（パッケージマネージャは pnpm のみ。npm/yarn を書かない）
- 配置は `utils/`（再利用ユーティリティ）か `src/`（features アーキテクチャ。routes → features → components → utils の単方向依存）かを明記する
- コード規約・コンポーネント構成は `doc/templates/coding.md` / `doc/templates/component.md` を参照させる（仕様に転記しない）
- 段階的に進める作業はフェーズ分けする（実例: `doc/plans/migrate-to-tasks-vision.md` の「移行フェーズ 0–5」形式）

### 4. レビューゲート

仕様をユーザーに提示し、明示的な承認を得る。承認前にコードを書かない。指摘があれば仕様を直して再提示する。

### 5. 保存して引き継ぐ

- 承認された仕様を `doc/plans/<topic>.md` に保存する（kebab-case。例: `benchmark-vite8.md`）
- タスク分解へ: `.agents/skills/planning-and-task-breakdown/SKILL.md`
- 実装中に前提が変わったら、コードより先に仕様を更新する（仕様は生きたドキュメント）

## Hard rules

- **仕様なしで非自明な実装を始めない**。複数ファイルにまたがる・アーキテクチャ判断を含む・30 分超の作業は全て対象
- **受け入れ基準に検証不能な項目を書かない**。「使いやすい」「きれいに」「ちゃんと」は基準ではない。コマンドか操作手順に変換できないなら、ユーザーに変換を依頼する
- 仮定を晒さずに仕様を書かない。曖昧・セキュリティセンシティブな要件はエスカレーションする
- ユーザー承認前に実装に入らない
- 検証コマンドは pnpm 表記のみ
- 出力本文は日本語。識別子・コマンド・パスは原文のまま

## 出力フォーマット

`doc/plans/<topic>.md` として保存する Markdown。既存ファイル（`benchmark-vite8.md` 等）と同様、見出し + 表中心で書く。

````markdown
# <トピック名>

## 背景・目的
<現状の問題と、この変更で何が良くなるか。2–4 行>

## 受け入れ基準
- [ ] <検証コマンド or 操作手順> が <期待結果> になる
- [ ] `pnpm vitest run` が green（新規挙動のテストを含む）
- [ ] `pnpm run lint` / `pnpm run build` が通る

## スコープ / 非スコープ
- やる: <...>
- やらない: <...>

## 前提・仮定
- <配置先（utils/ or src/）、依存方針、インターフェース維持の有無 等>

## 未解決の質問
- <実装前にユーザー判断が必要な項目。なければ「なし」>
````

## 完了チェックリスト

- [ ] 仮定リストを提示し、ユーザーの反応を得た（80% 未満の確信度は質問に変えた）
- [ ] すべての受け入れ基準が実行可能な検証（コマンド or 操作手順）で書かれている
- [ ] 非スコープが明記されている
- [ ] ユーザーが仕様を明示的に承認した
- [ ] 仕様を `doc/plans/<topic>.md` に保存した
- [ ] `.agents/skills/planning-and-task-breakdown/SKILL.md` へ引き継いだ（または単純な作業として直接実装する合意を得た）

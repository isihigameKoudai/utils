# Test-Driven Development

## ゴール

- すべての挙動変更が「先に失敗したテスト」によって証明された状態で完了する
- バグ修正は必ず「修正前に失敗する再現テスト」を伴う

## このリポジトリでのコマンド

```bash
pnpm vitest run                              # 全テスト 1 回実行(報告前は必ずこれ)
pnpm vitest run utils/array/array.test.ts    # 単一ファイル
pnpm vitest run -t "splitMap"                # テスト名パターン一致
pnpm run test                                # watch モード(開発中のみ)
```

- テストは実装と同じディレクトリに `*.test.ts` で置く(`array.ts` → `array.test.ts`)
- テスト名(`it`)は日本語で挙動を記述する(例: `it('配列を指定した数で分割する')`)
- すべての分岐とエッジケース(空配列・`undefined`・境界値)をカバーする(AGENTS.md の方針)

## 手順

### 1. RED — 失敗するテストを先に書く

1. 対象の挙動を 1 つ選び、テストを書く
2. 実行して **期待した理由で** 失敗することを確認する
   - assertion での失敗 → OK
   - import エラー・型エラーでの失敗 → テストがまだ対象に届いていない。テスト自体を直す
3. 最初から pass した場合、そのテストは何も証明していない。assertion を見直す

### 2. GREEN — 最小実装

- テストを通す最小のコードを書く。先回りの抽象化・オプション追加をしない
- `pnpm vitest run <対象ファイル>` で pass を確認

### 3. REFACTOR — green のまま整理

- 重複除去・命名改善・分割(ファイルは 300 行以内)
- 1 ステップごとにテストを再実行する

## バグ修正(Prove-It)

修正から着手しない。順序は固定:

1. バグを再現するテストを書く → 実行 → **失敗を確認**(失敗しないなら再現できていない。原因調査に戻る)
2. 修正を実装 → 対象テストが pass
3. `pnpm vitest run` で全件回帰確認

## テスト設計の規律

| 規律 | 内容 |
|---|---|
| 状態を検証する | 操作の結果(返り値・状態)を assert する。内部の呼び出し順・呼び出し回数を assert しない(リファクタで壊れる) |
| mock は最終手段 | 実物 > fake > stub > mock の順で選ぶ。`vi.mock` / `vi.spyOn` は外部 API・非決定処理(時刻・乱数)に限る |
| 1 テスト 1 概念 | Arrange-Act-Assert で構成し、複数の挙動を 1 つの `it` に詰めない |
| DAMP > DRY | 各テストが単体で読めること。共有ヘルパーへの過度な畳み込みをしない |

## Hard rules

- テストなしで挙動変更を完了としない
- 「全テスト pass」と報告する前に、必ず `pnpm vitest run` を実行し実際の出力を確認する
- 通すために無関係なテストを skip・削除・改変しない
- コードに変更がないのに同じテストコマンドを繰り返さない(確認は 1 回で足りる)
- ブラウザでしか確認できない挙動(描画・操作)は単体テストで代替しない → `.agents/skills/browser-testing-with-devtools/SKILL.md`

## 完了チェックリスト

- [ ] 新しい挙動それぞれに対応するテストがある
- [ ] バグ修正には「修正前に失敗した」再現テストがある
- [ ] `pnpm vitest run` が green(出力を確認した)
- [ ] `pnpm run lint` と `pnpm run build` が通る
- [ ] skip・無効化したテストがない
